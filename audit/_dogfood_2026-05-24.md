## New bugs the static audit missed

### B1. Top nav text collision at studio width (every page)

`Page.tsx` chrome packs the section nav (HEM/ÖVNING/LEKTION/FEEDBACK/FRAMSTEG) AND the tweaks-strip (light/dark, palette × 4, edition × 3, "egen" status) into a single header row with no wrap/hide rule.

Measured at vw=1265:
- Nav links span `left: 388 → right: 741`
- Palette+edition controls start at `left: 611`, end at `left: 1062`
- `Feedback` (592-657) is overlapped by `sand` button (633-664) and the rest of the palette row

Result: header text reads as `FE_DBACK · M_R_M_S_T_E_G · Grose · editorial workbook cockpit`. First thing on every page; reads broken.

**Where to fix**: `Page.tsx:243-280` (the header band). Either wrap the tweaks row below the nav at this breakpoint, or move it into a popover behind a single "⋯" affordance.

### B2. Breadcrumb separator missing on /lektion

Picker shows `← HEMLEKTION`. Reader shows `← ALLA SEKTIONERLEKTION`. Two breadcrumb tokens concatenated without separator.

**Where to fix**: `routes/lektion.tsx` — the `←` link template (search for the `Hem` / `Alla sektioner` tokens).

### B3. Status footer label inconsistent across session-template routes

Footer renders `-- ÖVNING -- redo` on:
- `/repetition` (should be `REPETITION`)
- `/diagnostik` (should be `DIAGNOSTIK`)

Caused by `SessionPlayer` passing a hardcoded `status.mode = 'ÖVNING'` to its `Page` child regardless of `sessionKind`.

**Where to fix**: `components/session/SessionPlayer.tsx` — derive `status.mode` from `sectionsProp` or `sessionKind`.

### B4. Running head English on /diagnostik

Chrome shows `HP · COACH · DIAGNOSTIC` (English) above a Swedish body eyebrow `DIAGNOS`. CLAUDE.md prohibits English in product strings except for ELF section content.

**Where to fix**: `routes/diagnostik.tsx` — the `runningHead` prop passed to `Page`. Replace `DIAGNOSTIC` → `DIAGNOSTIK`.

### B5. Running head shows section, not "REPETITION", on /repetition

Chrome shows `HP · COACH · ORD` on `/repetition` (because ORD is the section of the first queued question). The user is on the repetition queue *picker*, not an ORD drill.

**Where to fix**: `routes/repetition.tsx` — `runningHead` should be a fixed `Repetition` string while in the idle phase.

### B6. Drill session start silently fails — TWO underlying causes

Clicking `Starta övning →` on `/drill?section=KVA` does NOT transition to question phase. The page stays on idle. Browser console:

```
Uncaught (in promise) SyntaxError: Unexpected token '<', "<!doctype "... is not valid JSON
```

Investigated and split into two distinct bugs.

#### B6a — `var-2026.json` stale in Vite's static-asset map (PRIMARY)

The drill picker calls `pickDrillQuestions(section)` → `loadBank()` → fetches all 27 exam JSON files. 26 of them return `Content-Type: application/json`. **Just `var-2026.json` returns `text/html` (the SPA fallback `index.html`)** with etag `W/"779-..."` — same etag as `/foo` or any nonexistent path. Worker is running. API config is correct. The file exists on disk (304074 bytes) and is valid JSON.

Cause: Vite's static asset map registered `var-2026.json` as "not present" at some moment (before the file was added to public/data/). The file watcher fires `page reload` events but doesn't re-register the file as a static asset. Vite has been running since 2026-05-19 13:51:23.

Fix during this session: `rm var-2026.json && cp /tmp/var-2026-backup.json var-2026.json` — Vite re-registered it. Now serves correctly.

**Lasting fix to ship:**
1. Restart `pnpm dev` whenever new files appear in `public/`.
2. Add a smoke check to `scripts/` that hits every `public/data/*.json` URL and asserts `Content-Type: application/json` — catches this regression.
3. Long-term: the `app/scripts/sync-dataset.sh` script that populates `public/data/` should signal Vite to invalidate its asset cache after writes (e.g. by touching `vite.config.ts` or running a Vite cache-clear).

#### B6b — `/drill?section=X` flips to DIREKTLÄNK mode after Starta click (SECONDARY)

After B6a was fixed, clicking `Starta övning →` from `/drill?section=NOG` *does* create a session server-side and update the URL to `/drill?section=NOG&qid=var-2013-kvant1-NOG-024`. But the visible page then renders the `DIREKTLÄNK` landing screen instead of the question.

Root cause: `routes/drill.tsx:140-144`:

```ts
const pickQuestions = qid
  ? () => loadBank().then((b) => [findQuestion(b, qid)])
  : framework
    ? () => pickFrameworkQuestions(section as Section, framework)
    : () => pickDrillQuestions(section as Section, DEFAULT_DRILL_LENGTH)
```

And the matching idle-copy ternaries at lines 146-160 (`idleEyebrow = qid ? 'Direktlänk' : ...`).

When SessionPlayer's `urlSyncedQid.setQid(activeQid)` adds `?qid=` to the URL post-session-start, the route re-evaluates with `qid` set, swaps `pickQuestions` to the single-question picker, and renders the direct-link idle copy. The user is permanently stuck — clicking Starta again does the same thing.

This is a regression from PR #146 (URL-as-state). The intent was *resume-after-refresh*, but `?qid=` short-circuits drill-mode any time both `?section=` and `?qid=` are present.

**Where to fix**: `app/src/routes/drill.tsx:140-144`. Two viable approaches:

Option A — distinguish initial-qid vs setQid-mid-session via `useActiveSession()`:

```ts
const { data: activeSession } = useActiveSession()
const directLinkMode = !!qid && (!activeSession || activeSession.sections !== section)
```

Mirror that into the `idleEyebrow / idleHeadline / idleSubcopy / idleMeta / emptyCopy / pickQuestions` ternaries.

Option B — capture the initial qid in a `useRef` once and ignore further URL changes for the direct-link decision:

```ts
const initialQidRef = useRef(qid)
const directLinkMode = !!initialQidRef.current
```

Option B is simpler but loses the "deep-link to a specific question via URL share" case. Option A preserves it.

### B7. Home content invisible during reveal (520ms)

First-paint screenshot of `/` caught date+score+footer rendered with everything else (greeting, traps card, plan card) at `opacity: 0` mid-animation. After ~520ms the reveal completes and content appears. If a slow load compounds with this animation, the user can stare at a near-blank home for over a second.

**Where to fix**: `index.css @keyframes hpc-reveal` — keep the animation but start at higher opacity (e.g. 0.6) so the "missing content" feeling never appears.

### B8. Sticky footer overlaps section-table rows on /progress

The `Page.tsx` footer is `position: sticky; bottom: 0`. On `/progress`, mid-scroll, the floating `-- FRAMSTEG -- översikt` bar covers the NOG row in the SEKTIONER table.

**Where to fix**: add `padding-bottom: var(--footer-height)` to the `Page` scroll container, or back off `sticky` in favor of `position: fixed` with a layout slot reserved.

### B9. `tweaks dev` + `share` buttons always visible bottom-right

Confirmed: both ship in production-style builds. `ShareDebugButton` is mounted in `__root.tsx:57` without an `import.meta.env.DEV` guard. The `tweaks` launcher is gated by a session flag but the button still renders.

### B10. Cmd+K palette exposes `Dev panel /DEV` directly to users

Confirmed in production palette. 15 static commands include a `Dev panel /DEV` entry that routes to `/dev`. Should be filtered when not in dev.
